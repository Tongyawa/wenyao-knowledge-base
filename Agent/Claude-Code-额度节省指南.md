# Claude Code 额度节省指南

> 来源:与 Claude 的一次问答会话整理。
> 范围:订阅额度(token)消耗与 prompt 缓存机制。
> 标注约定:**[API 确定]** = Anthropic API 层文档化/可确认的行为;**[CC 不确定]** = Claude Code 客户端内部实现,无法从对话侧确认,但受同一套 API 缓存规则约束,故操作结论一致。

---

## 0. 底层第一性原理

**缓存是「请求(request)」的属性,不是客户端 UI 状态的属性。** [API 确定]

- 缓存条目存在 Anthropic 服务端,**只有真实 API 请求发出时**,服务端才拿这条请求的「前缀字节 + 相关参数」去比对已有缓存,决定命中还是重建。
- 没发消息 = 没有请求 = 服务端什么都没发生。在客户端拨设置只是改本地待发参数。
- 缓存命中把「重复读同一前缀」降到约 **0.1×**,但 **input tokens 永远不免费**。
- 省额度 = **少换前缀 + 控 context 体积 + 别空转 TTL**。

---

## 1. 缓存的成本与寿命 [API 确定]

| 项 | 数值 |
|---|---|
| 读命中 | ≈ 0.1× 基础 input 价 |
| 写入(5 分钟 TTL,默认) | ≈ 1.25× |
| 写入(1 小时 TTL,可选) | ≈ 2× |

**TTL 是滑动窗口,不是从写入硬性倒计时:**

- 默认 **5 分钟**,每命中一次就**重置续期**。
- 只要在窗口内持续命中,缓存实际上一直存活;连续活跃的会话里前缀缓存可以撑很久。
- **闲置**超过窗口(默认 5 分钟无任何命中)→ 被驱逐 → 下一条消息冷写。
- 过期是「时间到了」,与你切不切设置无关。
- 无手动清缓存 / 手动续期开关;续期只能靠「窗口内再发一条命中的请求」。

但注意：桌面端实测并非一定是5min，有时候半小时都能接上hit，所以只能说尽可能连续，但不用强迫自己5min内回复。

这也解释了：如果你每次都是发一条就干别的半天才回来看，max5x都感觉很不够用，甚至可能跟别人一个小小pro用着差不多

> 注:缓存 TTL **不是桌面端可见/可调的设置项**。它由 Claude Code 发请求时代码里的 `cache_control` 决定,不暴露成 UI 开关。想知道实际是 5m 还是 1h 只能实测(发消息建缓存 → 静置 → 再发看是否仍命中)。

## 2. 会掉缓存的动作(按破坏力排序)

| 动作 | 影响 | 省法 |
|---|---|---|
| **换模型**(含被 flag 强制切换) | **三层全冷**(tools/system/messages),最贵 | 少换;要切回趁 TTL 内;考虑关掉 flag 自动切模型 |
| 改工具集 / MCP 开关 | 全冷(工具在前缀最前端) | 别中途增删 MCP / 工具 |
| 改 CLAUDE.md / 用户偏好(若被热重载) | system + messages 失效 | 改完别在当前会话强制重载;让它在下个新会话生效 |
| 切思考强度后**发消息** | 可能重算 messages 层;tools+system 仍热 | 一段时间内固定一档,别逐轮 churn |
| compact | 新(压缩后)前缀一次冷写,靠后续多轮摊薄 | 只在后面还有多轮时做;收尾别做 |

**前缀渲染顺序:`tools` → `system` → `messages`。** 任何一处靠前的字节变化,会让其后所有内容失效。所以「换模型 / 改工具」最贵(动到最前端),「改消息内容」最轻(只动最后)。

---

## 3. 不掉缓存 / 免费的动作

- **纯切设置不发消息**:思考强度 A→B→A,只要不发、或最终落回原档再发 → **零成本**。掉不掉只在「按下发送」那一刻结算,依据是那条请求的参数;来回切但最终落回原档再发 = 等于没切。 [API 确定]
- **rewind 回到仍在 TTL 内的同模型旧点** → 重新命中那段老缓存,被丢弃的后续内容只是不再重放;还能借此「甩掉」之前的冷读分支。 [API 确定 + CC 不确定:依赖客户端保留同一前缀字节]
- **改用户偏好 / CLAUDE.md** → 对运行中会话默认不生效也不掉缓存(磁盘改动不回灌进已拼好的会话前缀);新会话冷启动本就要写一遍前缀,这条偏好顺带搭车 = **免费**。 [CC 不确定:是否热重载 / 何时重读文件无法确认,但两种分支——不重载=不掉不生效;重载=生效但掉一次缓存——必居其一,不存在「又生效又不掉」]

## 4. 重点场景拆解

### 4.1 被 Anthropic 强制切换模型(如 Fable5 → Opus4.8)

缓存按模型隔离,**换模型 = 三层全失效**。 [API 确定]

- **被强切的那条**:跑在新模型上,该模型对本会话无任何缓存 → 整段 transcript **全价冷读一遍** + 在新模型下写新缓存。一次实打实的肉疼。
- **下一条手动切回原模型**:又一次换模型冷读。但**关键省法**——原模型(Fable5)的老缓存若还在 TTL 内,它覆盖「切换点之前」的前缀仍有效;切回时**那段老历史能命中老缓存**,只有中间插入的那轮新模型对话是冷增量。
- **所以:要切回就趁早(5 分钟 TTL 内)**;拖过窗口老缓存被驱逐,整段重冷读。
- **杠杆**:桌面端 `Switch models when a message is flagged` 开关。关掉它 → 被 flag 时**会话暂停而非切模型**,避免「强切 = 整段冷读」(代价:那一轮无法继续)。Fable5 临下线、怕频繁强切烧额度时,这是能直接控制的阀门。

### 4.2 rewind

rewind 本身**不删缓存**。它把对话回退到更早的点,那段前缀字节没变;只要还在 TTL 内 + 同模型,继续发 → 重新命中。

- **rewind 回到模型切换之前(仍在原模型那条)并继续用原模型** → 前缀匹配老缓存,TTL 内命中。这等于甩掉后面那次冷读,不让它拖进后续上下文。
- 回退点缓存已过 TTL → 冷读。

### 4.3 compact 的时机

**缓存只在真实请求发生时被烧写,没有后台空写。** [API 确定]

- compact **不会**点完后凭空先写好缓存等着;新的压缩后前缀,是在「下一条带着它的请求发出去时」才被冷写入,也从那刻起后续才吃得到它的命中。
- compact 的成本 =「**一次新前缀冷写**」,靠**之后多轮**摊薄回本。
- **收尾一两条消息:别先 compact,直接发。**
  - 先发(不 compact):每条吃现成热缓存(~0.1×)+ 小增量,无冷重建。
  - 先 compact 再发 1–2 条:白付一次新前缀冷写,只剩 1–2 轮摊薄 → 回不了本,纯亏。
- compact 是「投资」,只在**后面还要长聊**时划算。
- 同一段上下文,compact 结果可能随思考强度变(若压缩调用吸收当前 effort)或不变(若用独立固定配置)——这点 [CC 不确定];且即便 effort 相同,摘要因采样随机性也非逐字一致。

## 5. 操作习惯清单

1. **能不换模型就不换**;被强切后要回去就**立刻**回(趁老缓存热),别来回弹。
2. **批量、连续地发**:在 TTL 窗口内保持节奏,别让缓存空转过期;长时间离开 = 回来默认冷读。
3. **收尾阶段直接发,不 compact**;compact 留给「上下文快爆 + 后面还要长聊」。
4. **控 context 体积**:精简 CLAUDE.md、只挂必要 MCP/工具——这是每轮额度的地板,降它比抠缓存更根本。
5. **设置类调整(model / 工具 / 偏好)集中在新会话前做**,别在长会话中途改去触发整段重算。
6. **思考强度一段时间固定一档**,别逐轮 churn;纯试探性拨档只要不发消息就免费。

---

## 附:确定性边界说明

- **有把握(API 层)**:前缀逐字节匹配、缓存按模型隔离、TTL 滑动续期约 5 分钟、读写倍率、缓存只在请求时结算。
- **无法从对话侧确认(Claude Code 客户端实现)**:fork / rewind / 强制切换后的缓存复用细节、compact 的确切写入时点、CLAUDE.md 是否及何时热重载、思考强度是否喂入 compaction。
- 但上述客户端行为都受同一套 API 缓存规则约束,因此本指南的**操作结论在两种实现下一致**。
